# Naming Conventions

Naming should optimize for recognition and predictability.

## Repository Names

Repository names should be:

- Clear rather than clever.
- Specific enough to distinguish purpose.
- Consistent with related repositories when part of a wider ecosystem.

Prefer names that indicate the project itself, not only an internal codename.

## File and Folder Names

Use names that communicate function.

Examples:

- `docs/` for reference material.
- `examples/` for sample implementations.
- `data/` for datasets or structured project data.
- `src/` for primary source code.

Avoid inconsistent abbreviations unless they are already standard in the relevant ecosystem.

## Consistency Across a Family of Repositories

When multiple repositories are related, naming should make that relationship visible. Shared prefixes, suffixes, or parallel naming patterns can reduce confusion and improve discoverability.
